# KV Cache: The Invisible Trick Behind Every LLM

[0:00] Same prompt, same model. The first call costs a dollar, the second cost 5 cents, 20 times cheaper. The reason is one of the most important tricks in modern AI inference. And the underlying idea is older than the transformer itself. Before 2017, language models process sentences one word at a time, slow to train. Then a paper called attention is all you need. Rewired everything. The trick. Every word could look at every other word in parallel all at once. training got dramatically faster. But there was a catch. Inference, which is what happens when the model is generating responses for you, is still sequential. Each new word depends on the words before it. So at inference time, transformers lose the parallelism they were designed for. First piece of vocabulary you need, a token. A token is a chunk of text, sometimes a whole word, sometimes a fragment. When you send a prompt, the model slices it into tokens before doing anything else. Second piece, the embedding. The model converts every token into a list of about 4,000 numbers called its embedding. Where do those numbers come from? They sit inside a giant table stored in the model file.

[1:09] One entry per word in the vocabulary. The values were learned during pre-training, the original training run that built the model. So inside the model, the word cat is not text. Its a list of 4,000 numbers. Third piece, and this one matters most, the dot product. Picture an embedding as an arrow in space. An arrow with about 4,000 dimensions, but for visualization, lets flatten to two. The dotproduct of two arrows gives you one number. Aligned arrows are positive and big. Perpendicular ones give zero, and opposite ones give negative numbers. Now, heres the beautiful part. During pre-training, the model nudged its embeddings until related concepts ended up pointed in similar directions. So cat and kitten point the same way. Cat and yacht dont. That means asking how related two words are reduces to checking how aligned their arrows are. One operation captures meaning. Search engines have used vector similarity since the 70s.

[2:06] Now back to attention. For each token, the model computes two new vectors from its embedding. Why two? Because each token plays two roles. What it advertises if you look at it. Thats the key. what it actually contributes if you pick it. Thats the value. Splitting the tokens role this way lets the model learn to advertise differently than it contributes. How are key and value computed? Multiply the embedding by a matrix. A matrix is a grid of train numbers also stored in the model file. The model has two of these matrices for attention. One produces the keys, the other produces the values. Run the embedding through the first matrix and you get the key vector. Run it through the second and you get the value vector. Both matrices were learned during pre-training alongside everything else. Walk it on a real prompt. Take the cat sat. Each word has its key and its value. To predict the next word, the model takes a new query and dots it with every prior key. Cat dot that gives 0.2 barely related. Cat dotted with cat itself scores 0.8 and cat dotted with satlands at 0.7. The values get weighted by their scores. sum them. That sum is the input for predicting the next word. Now, the hidden cost that matrix multiplication is expensive. Multiplying a 4,000dimensional embedding by a 4,000x4,000 grid is 16 million multiplications. And the model is built from 40 stacked layers like floors in a building. Each floor does its own attention. So, across all 40 layers, 600 million multiplications per token just to build the keys. The values cost the same. Setting up attention for one token costs roughly two billion math operations. Now generate the next token.

[3:49] The naive transformer recomputes K and V for the entire prefix with the same input running through the same matrices and producing the same outputs as last token. By a thousand tokens deep, you have redone the same K for the word the about a thousand times. Hundreds of billions of identical multiplications. Computer scientists noticed this kind of waste back in 1968. Donald Michie called the fixed memorization. Save the answer. Do not compute it twice. Memorization applied to attention is called the KV cache. It is a stack of saved keys and values from prior tokens kept in GPU memory. After each token, save its K and V. Add them to the stack. Next token, only compute the new pair. Run attention against the full stack. Why cache K and V instead of just the embedding? Because embeddings are cheap. The expensive thing is the matrix multiplication that turns embeddings into K and V cache. The expensive intermediate, not the input. Without this trick, generation would cost roughly a thousand times more compute.

[4:50] But that cache lives in GPU memory. And every token you generate adds another entry. For a typical 70 billion parameter model, thats about half a megabyte per token. Generate 100,000 tokens. You are carrying 50 GB of cache. Each concurrent user has their own. And here is the thing computer architects warned about back in 1995. Compute keeps doubling. Memory bandwidth does not. Wulf and McKee called it the memory wall. Your inference provider is not selling you compute. They are selling you GPU memory bandwidth. Now, back to your bill. Anthropic in 2024 made the cache reusable across API calls. They called it prompt caching. send the same prefix as a recent request and the provider keeps your KV cache in GPU memory and just runs attention against it. No matrix multiplications, no setup cost. The first call paid for the cache, the second call rents it.

[5:42] That is why your second call cost 5 cents instead of a dollar. And here is the math. Reason order matters. When you prompt, put stable stuff at the bottom. System prompt tools retrieve documents. Put the users actual question at the end. Why? Because the keys and values at later positions depend on the models processing of every earlier position through every layer. Change anything in the prefix and every cache entry from that point up is invalid. Move the user query to the front and you blow away your cache every request. Tail-load it and the cache reuses for free. Same exact tokens, different ordering, 10 times cheaper inference on agent loops. Long context is not slow because the model thinks harder.
